Autodebet Banking Architecture Pack

Fitur Autodebet untuk Mobile Banking, Internet Banking, dan Core Banking

Dokumen ini dirancang sebagai presentasi dan dashboard implementasi end-to-end. Definisi Singkat

Autodebet adalah fasilitas yang memberi otorisasi kepada bank untuk mendebit rekening nasabah secara otomatis berdasarkan mandate yang disetujui, pada jadwal tertentu atau saat tagihan tersedia, dengan tujuan pembayaran tagihan, cicilan, iuran, top up, transfer berkala, atau kewajiban lain.

  • Mobile / Internet Banking berperan untuk registrasi, persetujuan, maintenance, histori, dan notifikasi.
  • Middleware / API Layer berperan untuk validasi, orkestrasi, scheduler, routing, callback, dan retry.
  • Core Banking berperan untuk hold atau fund check, posting debit, ledger entry, reversal, fee posting, dan statement.
Business Scope Customer Journey

Cakupan Fitur per Kanal

Mobile Banking

Registrasi mandate, OTP atau biometric approval, ubah limit, suspend, resume, cancel, histori debit, push notification, dan dispute initiation.

Internet Banking

Fungsi setara mobile dengan tambahan tampilan rinci mandate, export histori, approval maker-checker untuk nasabah bisnis, dan dashboard tagihan.

Core Banking

Pembentukan standing instruction record, pengecekan saldo, debit account posting, fee posting, exception queue, reversal, EOD reports, dan general ledger impact.

Dari sisi produk, autodebet dapat dibagi menjadi beberapa variasi: fixed amount recurring, bill-based variable amount, minimum amount due, partial debit with retry, dan event-triggered debit. Masing-masing membutuhkan parameter yang berbeda pada mandate dan scheduler.

Primary Actors

Aktor Utama

  • Nasabah pemilik rekening sumber dana.
  • Channel aplikasi mobile banking atau internet banking.
  • Identity & authentication service untuk login, OTP, device binding, dan biometrik.
  • Autodebet orchestration service sebagai pusat business rule.
  • Biller, merchant, loan system, atau host tujuan penagihan.
  • Core banking dan general ledger untuk posting akuntansi.
  • Notification, reconciliation, fraud monitoring, dan customer service tools.
End-to-End Process Lifecycle

Flow Proses End-to-End Autodebet

1. Discovery Nasabah memilih menu autodebet, biller, sumber dana, dan tipe pembayaran.
2. Consent Nasabah memberikan persetujuan digital dan otentikasi kuat.
3. Validation Validasi rekening, status CIF, limit, legal terms, dan biller eligibility.
4. Mandate Setup Mandate disimpan dan jadwal aktif dibentuk pada orchestration atau core.
5. Triggering Scheduler atau biller callback memicu proses penagihan.
6. Debit Posting Core banking melakukan balance check, posting debit, fee, dan referensi jurnal.
7. Advice Status berhasil atau gagal dikirim ke biller, channel, notifikasi, dan histori.
8. Reconcile Settlement, rekonsiliasi, retry, reversal, dispute, dan pelaporan audit.

Rincian Flow Operasional per Tahap

Fase Aktivitas Utama Kontrol Sistem Output
Registrasi Nasabah memilih biller, rekening sumber, nominal atau rule tagihan, tanggal efektif, dan frekuensi. Validasi kepemilikan rekening, status rekening aktif, channel entitlement, dan pre-check biller. Draft mandate.

Maintenance Nasabah dapat suspend, resume, ubah limit, ganti rekening, atau terminasi mandate. Version control, effective date, immutable audit trail, approval step. Mandate lifecycle update.

Development Requirements

Kebutuhan Pengembangan Fungsional

  • Registrasi mandate baru, termasuk fixed amount dan variable bill amount.
  • Consent management dengan syarat & ketentuan yang terversioning.
  • Update mandate: nominal maksimum, tanggal debet, rekening sumber, suspend/resume, cancel.
  • Scheduler dengan calendar business day, holiday handling, cutoff time, dan retry rules.
  • Biller inquiry dan bill presentment untuk skenario amount variable.
  • Debit posting, fee posting, reversal, dan callback status.
  • Histori transaksi dan histori perubahan mandate.
  • Notifikasi real-time dan reminder sebelum pendebetan bila diwajibkan kebijakan bisnis.
  • Rekonsiliasi, settlement, exception handling, dan manual repair workflow.
Non Functional

Kebutuhan Non-Fungsional

  • Availability tinggi dengan active-active atau active-standby, terutama pada scheduler dan API gateway.
  • Idempotency kuat untuk create mandate, execute debit, callback, dan reversal.
  • Latency rendah untuk operasi channel, serta throughput tinggi untuk batch debit massal.
  • Audit trail immutable untuk compliance dan investigasi dispute.
  • Encryption in transit dan at rest untuk data rekening, CIF, token, dan consent artifact.
  • Observability lengkap: logs, metrics, traces, business counters, dead-letter queue.
  • Backward compatibility API dan versioning saat ada perubahan rule bisnis.
  • Resilience terhadap duplicate message, race condition, clock drift, dan partial outage.
API Catalogue Open API Style

Kebutuhan Spesifikasi API

API Method Consumer Tujuan Catatan Teknis
/v1/autodebit/billers GET Mobile/IB Mengambil daftar biller dan capability. Cacheable, filter by category, return maintenance status.
/v1/autodebit/inquiry POST Mobile/IB Cek tagihan atau validasi customer reference. Support partner adapter, timeout, dan response normalization.
/v1/autodebit/mandates POST Mobile/IB Membuat mandate baru. Wajib idempotency key, consent token, request signature.
/v1/autodebit/mandates/{id} GET Mobile/IB/CS Detail mandate dan histori status. Role-based masking dan pagination histori.
/v1/autodebit/mandates/{id} PATCH Mobile/IB Update limit, jadwal, rekening, atau status. Gunakan version number untuk optimistic locking.
/v1/autodebit/mandates/{id}/cancel POST Mobile/IB Terminasi mandate. Perlu strong auth dan effective date handling.
/v1/autodebit/executions POST Scheduler/Biller Memicu eksekusi debit. Harus asynchronous, masuk queue, dan punya execution reference.
/v1/autodebit/executions/{id} GET Channel/Biller/Ops Mengambil status eksekusi. Status machine harus konsisten dan retriable.
/v1/autodebit/callbacks/biller POST Autodebet Service Kirim hasil ke partner atau biller. mTLS, HMAC signature, retry with backoff, deduplication.
/v1/autodebit/reconcile/files POST Ops/Settlement Upload hasil rekonsiliasi atau settlement. Checksum file, dual control, exception bucket.

Header & Konvensi API yang Direkomendasikan

  • Authorization: Bearer <token>
  • X-Request-Id untuk tracing end-to-end.
  • Idempotency-Key untuk create mandate dan execute debit.
  • X-Channel-Id, X-Device-Id, X-Correlation-Id.
  • X-Signature atau JWS untuk integrasi partner kritikal.
  • Response code bisnis dipisahkan dari HTTP code, misalnya businessCode dan businessMessage.
  • Semua API harus mendukung schema evolution tanpa mematahkan consumer lama.

Status dan Error Code Penting

Status Mandate

DRAFT, PENDING_AUTH, ACTIVE, SUSPENDED, PENDING_CANCEL, CANCELLED, EXPIRED.

Status Execution

QUEUED, IN_PROGRESS, SUCCESS, FAILED, RETRY_SCHEDULED, REVERSED, SETTLED.

  • AD001 duplicate mandate.
  • AD002 invalid biller reference.
  • AD003 insufficient fund.
  • AD004 account restricted or dormant.
  • AD005 cutoff passed atau holiday rule violation.
  • AD006 biller timeout atau callback unavailable.
  • AD007 debit already processed.

Security & Compliance Control Framework

Kontrol Keamanan dan Kepatuhan yang Wajib Dipertimbangkan

Strong Customer Authentication

Gunakan minimal 2 faktor untuk registrasi, perubahan kritikal, dan cancel mandate.

Consent Evidence

Simpan teks persetujuan, timestamp, auth factor, device info, dan terms version.

Data Protection

Masking nomor rekening di UI, tokenisasi bila memungkinkan, dan enkripsi data sensitif.

Fraud Controls

Deteksi device baru, perubahan perilaku, velocity check, blacklist, dan anomaly scoring.

Risiko

Pendebetan Ganda

Terjadi akibat retry yang tidak idempotent, callback duplikat, atau job overlap.

Mitigasi: idempotency key, unique execution reference, locking, reconciliation alarm.

Risiko

Salah Nominal / Salah Rekening

Berasal dari inquiry yang salah, parameter mapping yang keliru, atau data mandate tidak sinkron.

Mitigasi: preview confirmation, mandate versioning, maker-checker, validation matrix.

Risiko

Insufficient Fund

Menyebabkan gagal bayar dan complaint bila nasabah tidak mendapat notifikasi jelas.

Mitigasi: reminder, retry policy, partial debit rule, fallback account bila diizinkan.

Risiko

Consent Tidak Sah

Dokumen otorisasi tidak tersimpan lengkap sehingga rawan sengketa.

Mitigasi: legal text archive, digital signature, immutable consent store.

Risiko

Outage Biller

Biller atau partner tidak tersedia saat inquiry atau callback.

Mitigasi: queue, retry backoff, circuit breaker, status pending settlement.

Kontrol

Audit & Pelaporan

Seluruh perubahan mandate, eksekusi, reversal, dan settlement harus dapat ditelusuri.

Implementasi: centralized logging, SIEM, report harian, dan evidence repository.

Operational Model Runbook Ready

Kebutuhan Operasional, Monitoring, dan Support

1. Pre-Execution Monitoring

Monitor jumlah job yang akan jatuh tempo, readiness partner, antrian inquiry, dan health scheduler sebelum window debit dimulai.

2. Live Execution Control

Pantau success rate, fail rate, timeout, duplicate prevention, debit throughput per menit, dan rejected postings.

3. Reconciliation Window

Lakukan pencocokan transaksi host, callback, file settlement, suspense posting, dan mismatch aging.

4. Customer Support Closure

Sediakan inquiry screen bagi CS untuk melihat evidence consent, histori mandate, dan detail execution.

Dashboard Operasional

Mandate active count, due today, success ratio, retry backlog, callback backlog, unresolved exceptions, settlement mismatch.

Runbook Utama

Partner outage, retry storm, duplicate suspicion, reversal process, manual requeue, reconciliation mismatch, dan complaint handling.

SLA Internal

Response inquiry, debit completion, callback delivery, notifikasi ke nasabah, reversal turnaround, dispute resolution.

Testing Strategy

Pengujian yang Dibutuhkan

  • Unit test business rule untuk frekuensi, holiday policy, cutoff, fee, dan retry.
  • Integration test dengan core banking, inquiry host, biller callback, dan notification.
  • Contract test untuk semua API eksternal dan internal adapter.
  • Performance test untuk debit massal pada jam puncak.
  • Failover test scheduler, MQ, dan DB cluster.
  • Security test: authentication bypass, replay attack, signature tampering, privilege escalation.
  • User acceptance test untuk nasabah retail dan korporasi.
Deployment

Lingkungan dan Deployment yang Direkomendasikan

  • Environment minimal: DEV, SIT, UAT, PERF, PROD, dan DR site.
  • Blue-green atau rolling deployment pada orchestration dan API gateway.
  • Scheduler state sebaiknya terpusat dan mendukung failover leadership.
  • Gunakan MQ atau event bus untuk memisahkan trigger debit dari posting sinkron.
  • Database terpisah untuk mandate master dan event log bila volume besar.
  • Backup, retention, dan archive policy wajib mengikuti aturan audit internal.
Implementation Roadmap Phased Delivery

Usulan Tahapan Implementasi

Phase 1: MVP

Fixed recurring debit internal bank, create/cancel mandate, OTP approval, scheduler dasar, debit posting, notifikasi, dan histori sederhana.

Phase 2: Enhanced Billing

Bill inquiry eksternal, variable amount, retry policy, callback partner, reconciliation dashboard, dan customer support tooling.

Phase 3: Enterprise Controls

Maker-checker, bulk registration, corporate internet banking, advanced fraud rule, fee engine, partial debit, dan analytics.

Phase 4: Ecosystem Scale

Open API partner onboarding, self-service biller portal, multi-bank mandate, event streaming, dan predictive collection intelligence.

Kesimpulan: implementasi autodebet yang baik tidak hanya berupa fitur jadwal debit, tetapi sebuah kapabilitas end-to-end yang memadukan consent, orchestration, posting core banking, security, reconciliation, dan customer support traceability. Jika dibangun sebagai layanan terpisah dengan API yang jelas, fitur ini akan lebih mudah dikembangkan lintas channel, lebih aman diaudit, dan lebih siap diskalakan untuk kebutuhan retail maupun korporasi.

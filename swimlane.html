Step 01
Nasabah membuka menu autodebet dan memilih biller atau jenis kewajiban.
Step 02
Channel meminta daftar biller, kategori, capability, dan rule registrasi.
Step 03
Autodebet API mengembalikan daftar biller, tipe amount, frekuensi, cutoff, dan requirement consent.
Step 04
Nasabah mengisi rekening sumber, customer reference, tanggal debet, limit maksimum, dan rule retry.
Step 05
Channel mengirim inquiry validasi customer reference atau tagihan ke Autodebet API untuk diteruskan ke Biller Adapter.
Step 06
Biller Adapter melakukan customer validation atau bill inquiry ke host biller.
Step 07
Autodebet API menerima hasil inquiry, normalisasi response, cek eligibility, duplicate mandate, dan parameter limit.
Step 08
Channel menampilkan ringkasan mandate, syarat ketentuan, dan consent statement ke nasabah.
Step 09
Channel meminta strong authentication ke IAM: OTP, PIN, device binding, atau biometric challenge.
Step 10
IAM / OTP memverifikasi challenge, menghasilkan auth result dan consent proof reference.
Step 11
Channel submit create mandate dengan idempotency key, consent token, dan channel reference.
Step 12
Autodebet API menyimpan mandate master, consent evidence, next schedule, dan audit trail.
Step 13
Core Banking menerima standing instruction setup atau reference linkage bila dibutuhkan pada host.
Step 14
Notification service mengirim success advice: mandate aktif, next debit date, dan rule pendebetan.
Exception
Jika inquiry gagal, duplicate ditemukan, auth gagal, atau rekening tidak eligible, proses dihentikan dan channel menerima business code yang sesuai.

Sequence Diagram Detail: Eksekusi Pendebetan Berkala

Scheduler
Autodebet API
Biller Adapter
Risk / Rule
Core Banking
Settlement
Notif / Channel
Step 01
Scheduler memilih mandate due hari ini berdasarkan next_run_at, cutoff, holiday policy, dan status ACTIVE.
Step 02
Autodebet API membuat execution record status QUEUED dan melakukan lock pada mandate.
Step 03
Untuk bill-based mandate, API meminta latest bill inquiry atau bill presentment ke Biller Adapter.
Step 04
Autodebet API mengecek amount rule, max limit, duplicate bill ref, retry count, blacklist, dan anomaly rule.
Step 05
API mengirim debit posting request ke Core Banking dengan execution reference unik.
Step 06
Core Banking melakukan balance check, account status check, fee calculation, lalu atomic debit posting.
Step 07
Jika sukses, Core mengembalikan journal ref, posting time, fee amount, dan booking status.
Step 08
Autodebet API update execution status menjadi SUCCESS atau RETRY_SCHEDULED atau FAILED.
Step 09
API mengirim callback hasil debit ke Biller Adapter atau partner collection host.
Step 10
Settlement atau reconciliation menandai matched, pending, atau mismatch sesuai callback dan file settlement.
Step 11
Notification service dan channel history diperbarui: sukses, gagal dana tidak cukup, atau akan retry.
Exception A
Jika saldo tidak cukup, status menjadi FAILED atau RETRY_SCHEDULED sesuai retry policy dan nasabah diberi notifikasi.
Exception B
Jika callback biller gagal, transaksi debit tetap tersimpan, masuk pending advice queue, dan dikirim ulang dengan backoff.
Exception C
Jika mismatch settlement atau reversal diperlukan, ops melakukan manual review berdasarkan audit trail dan journal reference.

Design Notes

Catatan Teknis Penting untuk Membaca Diagram

  • Registrasi dan eksekusi sebaiknya dipisahkan sebagai dua domain flow agar monitoring dan troubleshooting lebih mudah.
  • Execution harus bersifat asynchronous walaupun inquiry tertentu mungkin sinkron.
  • Idempotency wajib hadir pada create mandate, execution trigger, callback partner, dan reversal.
  • Mandate versioning penting agar perubahan limit atau rekening sumber tidak merusak histori lama.
  • Callback failure ke biller tidak boleh otomatis dianggap gagal debit bila core posting sudah sukses.
  • Swimlane bisa dijadikan basis penyusunan SOP, RACI Matrix (Responsible, Accountable, Consulted and Informed), BRD, dan sequence API yang lebih teknis.
